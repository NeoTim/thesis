SRP and password salting

Posted on August 10, 2012 by tomchop
  • 0
    inShare

Today, Blizzard US released a statement declaring that their North American servers had been breached and that players’ personal info had possibly been compromised. Fortunately, they say, all passwords are stored using to the SRP (Secure Remote Password) protocol. According to Blizzard, this makes password recovery “extremely difficult”. Unfortunately, that is not really the case.

SRP is not meant to prevent password cracking

The SRP protocol is a zero-knowledge password proof. This means that using SRP, the prover (party knowing the password) can prove that he has knowledge of said password without the verifier (party verifying that the prover actually knows the password) obtaining any other information that that fact. More details about zero-knowledge proofs (the generalization of zero-knowledge password proofs) can be found on this nice Wikipedia page. ZKPP are useful when used over password-authenticated key agreements, where encryption keys are based on the knowledge of a secret password, without eavesdroppers gaining any additional information on said passwords. In this way, SRP provides a secure way to provide client authentication, regardless of their length or complexity.

Therefore, SRP stores password verifiers instead of the passwords themselves. There is an excellent rant about SRP that pretty much explains why it is not as secure as it sounds. To sum it up, passwords are “stored” as a concatenation of a salt s, and a value v where , and . v and s are the password verifier and salt, and g and N are implementation-defined values.

The verifier host knows for each user and, according to Thomas Wu (the creator of SRP), that is enough to start a bruteforce attack. As we can see, for a given username, x can be computed for every P and v can be calculated. This value can be compared to the one stored in the database, just like in a classic bruteforce scenario. SRP does protect from rainbow table attacks though, since each verifier is supposed to be unique.

Jeremy Spilman, author of the blogpost detailing the inner security mechanisms of SRP (from which this post is largely inspired – the credit goes to Jeremy), evaluates the feasibility of an attack on Blizzard’s SRP database:

A recent Intel benchmark shows performance of 1024-bit and 512-bit ME on their i7 -2600 CPU (from 2011). Based on these numbers, I would extrapolate that the attacker can probably run over 100k 256-bit ME’s per second, for each CPU core they dedicate to the attack. At this rate, for each machine dedicated to cracking these passwords, they can check 100,000 of their top passwords against 400,000 usernames, per day. Since the attack happened over 5 days ago, millions of users’ passwords have likely already been cracked.

This contradicts quite clearly Blizzard’s initial statement on the passwords being “extremely difficult to extract”.

My password needs more salt

This (and comments of people on twitter) made me realize how much salting passwords is overrated. The media sees this as a salvation for any company that uses it on their password database. However, it is important to note how salting passwords increases the security of the password database as a whole as opposed to each individual password.

Unix stores password hashes using hash functions, which are deterministic. This means that for a given input (e.g. a password), every run of the hash function will give you the same output. This means that everyone that uses ’123456′ as a password will have his password stored as ‘e10adc3949ba59abbe56e057f20f883e’ (which is the MD5 hash for the text ’123456′). If a large (500k~) userbase would come to be compromised, and no salts were used (did anyone say LinkedIn?) then the attackers could use pre-computed rainbow tables (huge tables listing all password – hash combinations for a given hash function: in our example, or classic bruteforce attacks. For each possible password combination, the hash is calculated and then compared to every entry in the compromised database. If 30k users are using ’123456′ as a password, then all 30k user’s accounts will be compromised simultaneously.

When salts are used, mass-compromising accounts becomes more difficult. Today, 128-bit salts are generated and stored for each individual password. Generally, when a user authenticates to a system, the system concatenates the salt and the provided password and hashes the result, and then compares it to the hash stored in the system. With 128-bit salts, you can get as much as different hashes for the same password. Even with today’s technology, generating rainbow tables with all possible values for every password remains a challenge.

What password salting is not

Password salting protects a database of passwords from being mass-compromised. If the 30k users using ’123456′ as a password have their passwords salted, then the hashes will all be different. But it is important to understand that salts are not cryptographically protected. They are stored right next to the password hashes in /etc/shadow (in Linux environments). If your system gents compromised, then so do your password hashes and so do the salts. Therefore, bruteforcing a single salted password using a classical dictionary attack is as complex as attacking an unsalted password.

Password salting does not increase password complexity. If you use a two-character password and a 128-bit salt, the attacker will still only have to guess the last two characters of your password in order to find a matching hash, since he knows the salt beforehand. All he has to do instead of hashing all two-character combination, is concatenate the salt before hashing.

Then how can I secure my passwords?

A final word concerning password storing and general account security. You can’t [really] blame a company for having their servers compromised. It can happen to anyone out there; governments are saying that a huge amount of companies are already unknowingly compromised, and that the ones that don’t will be soon. Blizzard did their job well in this case, by storing passwords in a more-or-less secure fashion. There was little more they could have done in order to mitigate the risks of password compromise following a successful breach of their servers.

What Blizzard did wrong, on the other hand, was to specify such a low-entropy password policy (Only lowercase letters and numbers allowed? Seriously?), and falsely stating that password extraction was very difficult. That is not true, and it is even less true knowing that all passwords are so weak.

This kind of password policy reminds me of something else. People need to start understanding that information security is a game that everyone (including them) has to play. It is the company’s responsibility to securely store passwords just as it is the user’s responsibility to choose it wisely. Using ‘password’ as a password is exactly the same as being naive enough to send 100 € to that poor heir from Senegal, or sending baking credentials to a phishing site. Healthy security habits should be taken by everyone: verifying that a connection is secure and authenticated, using strong and complex passwords (or passphrases), make sure to call your friend who is on holidays in Spain and really needs that money to be wired to Western Union, never using the same password on different sites, or never disclosing it.

Companies: don’t force users to choose weak passwords, you’re doing something stupid. I wouldn’t force them to use strong passwords either, as I think that choosing a password falls under their responsibility. Also, start communicating honestly about your mistakes: Blizzard tried to cover their asses saying that RDP makes password extraction near-impossible, which is false. This can induce people to thinking their password (simple as it is) will not be compromised and never change it. Just accept your part of responsibility in this and patch the hole that made the breach possible, and spend some money on pentesting (which is not the ultimate solution but just might have found a hole like the one that was exploited).

Users: Understand that a weak password will probably lead to someone hacking your account, and a strong password will probably prevent that. Stop relying on companies to do their jobs well (are you NOT reading the news lately?), and start taking measures to secure your stuff. Understand that you now you use computers not only when you check your email or go on Facebook, but every time: when you pay using your credit card, when you take the bus, when you watch TV… Treat your password as if it were your credit card PIN (I know people in France are VERY attached to it)
